# Маршрутизатор Конфликтов (Conflict Resolution Router)

В процессе `viewpoints` были выявлены кросс-функциональные конфликты. Решения нельзя отбирать просто голосованием, конфликты разрешаются через строгие методологические протоколы. Ваша задача классифицировать конфликт и направить к методу.

## Таксономия Конфликтов и Методов:
1. **Quality-Attribute Conflict** -> `ATAM` (Architecture Tradeoff Analysis Method).  
   *Когда применять:* Безопасность конфликтует с Производительностью. Сложные технические компромиссы.
2. **Alternative Ranking Conflict** -> `MCDA` (Multi-Criteria Decision Analysis).  
   *Когда применять:* Спор о том, какой вариант лучше подходит под матрицу весов (цена vs скорость внедрения).
3. **Stakeholder Conflict** -> `WinWin`.  
   *Когда применять:* Конфликт интересов между Отделами/Владельцами процессов на уровне политик.
4. **Uncertainty Conflict** -> `Delphi`.  
   *Когда применять:* Нет объективных данных, нужно экспертное сведение вероятностей или гадание на низких доверительных данных.
5. **Scope-Set Conflict** -> `SBCE` (Set-Based Concurrent Engineering).  
   *Когда применять:* Спор о выборе архитектуры, когда мы не можем позволить себе ошибиться на раннем этапе. Оставляем 2 опции жить параллельно до появления доказательств.

## Инструкции:
Прочитайте `viewpoint_conflicts` и `tradeoff_table`. 
Для каждого конфликта сформируйте запись в `ConflictRecords.md`, указав:
- `conflict_type`: Классификация по указанному выше словарю.
- `layers_involved`: Какие слои затронуты (например, Requirements, Functional).
- `method_selected`: Имя метода (ATAM/MCDA/etc).
- `selection_rationale`: Обоснование (почему этот метод эффективен здесь).
- `resolution_status`: routed.
