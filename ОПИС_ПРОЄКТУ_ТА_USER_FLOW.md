# Опис проєкту та User Flow

## 1. Суть проєкту

`electronic_consultant_v3` — це сервіс для структурованого розбору складних управлінських, операційних та організаційних кейсів. Його задача не просто "узагальнити текст", а провести користувацький кейс через послідовний аналітичний конвеєр і на виході дати:

- стислий управлінський висновок для керівника;
- повний аналітичний звіт з логікою, причинами, обмеженнями та варіантами рішень;
- трасування від початкового опису проблеми до рекомендованого рішення;
- артефакти, які можна використати для подальшого проєктування рішень, пілоту та архітектури.

По суті, це "електронний консультант", який допомагає розкласти неструктурований опис ситуації на формалізовані етапи аналізу: від вхідного кейсу до рішення, плану впровадження, відкату, перевірки ефекту та звітності.

## 2. Яку цінність отримує користувач

Користувач отримує не просто відповідь моделі, а повноцінний процес аналізу:

- виявлення реальної проблеми, а не лише симптомів;
- декомпозицію ситуації по шарах системи;
- аналіз з різних точок зору;
- формування набору альтернатив;
- вибір рішення через обмеження, компроміси та критерії прийняття;
- підготовку матеріалу для управлінського рішення.

Це особливо корисно там, де є:

- вузькі місця в бізнес-процесах;
- конфлікт між швидкістю, якістю та витратами;
- потреба пояснити керівнику, що саме зламалося і чому;
- потреба перейти від "інтуїтивного" розбору до відтворюваної логіки.

## 3. Загальна логіка роботи сервісу

Сервіс побудований як pipeline з кількох послідовних стадій.

### 3.1. Intake

Система приймає вхідний кейс: документ, текстовий опис, нотатку або файл від користувача. На цьому етапі кейс нормалізується і перетворюється на структурований вхід для подальшої роботи.

### 3.2. Layer Analysis

Кейс розкладається по кількох системних шарах:

- бізнес-логіка;
- вимоги та обмеження;
- функціональна модель;
- розподіл ролей і відповідальності.

Це дозволяє не плутати симптом, процес, рольову помилку та бізнес-наслідок.

### 3.3. Viewpoints

Проблема аналізується з кількох точок зору: стратегічної, аналітичної, операційної, архітектурної, критичної та клієнтської. Це зменшує ризик однобокого висновку.

### 3.4. Characterization

Система формує паспорт ситуації:

- цілі оптимізації;
- жорсткі обмеження;
- ризики;
- слабку ланку;
- ключові метрики, які треба контролювати.

### 3.5. Problem Factory

На цьому етапі система формулює саме проблему, яку потрібно вирішувати. Це важливо, бо в реальних кейсах користувач часто приносить не проблему, а симптом або емоційне трактування ситуації.

### 3.6. Solution Factory

Система генерує кілька альтернатив, порівнює їх, проводить parity/trade-off аналіз, розв’язує конфлікти між варіантами та формує рекомендоване рішення.

### 3.7. Decision & Operations

Після вибору рішення створюються:

- ADR або запис про архітектурне рішення;
- runbook для впровадження;
- rollback plan;
- умови повторної перевірки або re-entry.

### 3.8. Reporting

Наприкінці система готує два ключові звіти:

- `Executive_Summary.md` — для керівника;
- `Analytical_Full_Report.md` — для глибокого аналізу, команди або архітектора.

## 4. User Flow: як користувач взаємодіє з сервісом

Нижче наведено базовий сценарій взаємодії.

### Крок 1. Користувач готує кейс

Користувач має опис проблеми у вигляді:

- `.docx`
- `.doc`
- `.txt`
- `.md`

Це може бути опис бізнес-ситуації, конфлікту в процесі, управлінської проблеми, операційного збою або кейсу клієнта.

### Крок 2. Користувач запускає аналіз

Користувач запускає:

```bash
python3 run_case.py "/шлях/до/файлу"
```

Або покроково окремі стадії через скрипти в `scripts/`, якщо потрібен більш контрольований режим.

### Крок 3. Сервіс створює workspace кейсу

Для кожного запуску створюється окрема папка в `cases/`, де зберігаються всі артефакти аналізу. Це дає:

- ізоляцію кейсів;
- повторюваність;
- можливість аудиту;
- можливість повертатися до будь-якої стадії.

### Крок 4. Сервіс проходить аналітичний pipeline

Система автоматично проводить кейс через етапи:

- intake;
- layers;
- viewpoints;
- characterization;
- problem factory;
- solution factory;
- reporting.

На кожному етапі з’являються окремі артефакти, які можуть бути перевірені, використані повторно або переглянуті людиною.

### Крок 5. Користувач отримує результат

Після завершення аналізу користувач отримує:

- короткий управлінський звіт;
- повний аналітичний звіт;
- логіку вибору рішення;
- план впровадження;
- план відкату;
- матеріал для подальшої деталізації архітектури.

## 5. Що саме може робити користувач

Користувач сервісу має такі можливості:

- завантажити новий кейс на аналіз;
- запустити повний автоматичний розбір;
- запускати окремі стадії pipeline вручну;
- переглядати всі проміжні артефакти;
- повторно генерувати звіти;
- валідувати workspace;
- аналізувати, на якому етапі виник збій;
- використовувати результати для підготовки управлінського рішення або майбутньої архітектури рішення.

## 6. Яким має бути досвід користувача

З точки зору користувача сервіс має сприйматися так:

> Я описую проблему своєю мовою, завантажую файл у систему, запускаю аналіз і отримую не хаотичну відповідь ШІ, а структурований пакет висновків: що саме зламалося, чому це сталося, які є варіанти дії, що рекомендується робити та як це впроваджувати без неконтрольованого ризику.

Тобто користувач взаємодіє із сервісом не як з "чатом", а як з аналітичною системою підтримки рішень.

## 7. Основні ролі користувачів

Сервіс може бути корисним кільком типам користувачів.

### Керівник або власник бізнесу

Отримує короткий executive summary:

- що сталося;
- де проблема;
- чому це важливо;
- що рекомендується робити;
- які ризики та умови відкату.

### Аналітик або консультант

Отримує повний аналітичний ланцюг:

- від нормалізації кейсу;
- до формування проблеми;
- до вибору рішення;
- до обґрунтування через критерії та компроміси.

### Архітектор або системний проєктувальник

Може використовувати результати як основу для деталізації:

- цільової архітектури;
- операційної моделі;
- ролей;
- плану впровадження;
- критеріїв перевірки ефекту.

## 8. Як з цього виростає детальна архітектура

Цей сервіс не замінює детальне архітектурне проєктування, але створює для нього правильну базу.

Після проходження користувача через pipeline вже з’являються:

- чітко сформульована проблема;
- acceptance criteria;
- обмеження;
- shortlist рішень;
- обране рішення;
- логіка впровадження;
- rollback умови;
- метрики ефекту.

Саме з цього можна будувати:

- продуктову архітектуру;
- процесну архітектуру;
- ІТ-архітектуру;
- дорожню карту впровадження;
- систему управління ризиками та перевірки результату.

## 9. Коротко в одному абзаці

`electronic_consultant_v3` — це сервіс аналітичного розбору кейсів, який приймає неструктурований опис проблеми, проводить його через послідовний pipeline системного аналізу, формує проблему, генерує та порівнює варіанти рішень, обирає рекомендований сценарій, готує план впровадження та відкату і завершує все управлінським та аналітичним звітом. Для користувача це означає простий сценарій: завантажити кейс, запустити аналіз, отримати готовий пакет рішень та логіку, на основі якої вже можна проектувати детальну архітектуру.
