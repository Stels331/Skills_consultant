# S2-T1 — Intake + Parsing pipeline (глубокое описание)

## Мета
- Сформульована у задачі та узгоджена з цілями спринта.

## Що робимо
- Виконуємо обсяг робіт, описаний у цій задачі (див. робочі деталі нижче).

## Як виконання
- Реалізація по кроках із фіксацією артефактів у файловій структурі проекту.

## Очікуваний результат
- Отримано завершений артефакт задачі, готовий для інтеграції в потік спринта.

## Як тестувати виконання
- Запустити перевірки/скрипти модуля задачі локально в IDE.
- Перевірити, що всі вихідні файли задачі створені/оновлені коректно.
- Перевірити, що зміни не ламають попередній workflow спринта.

## Контрольні точки поза тестуванням
- Актуальність документації та читабельність артефактів для handoff.
- Відповідність архітектурним правилам (problem-first, evidence-first, no-DB v1).
- Відмітка прогресу задачі та синхронізація зі статусом спринта.

## Робочі деталі (початковий опис задачі)

**Приоритет:** Must


## Scope
- Поддерживаем типы входа: raw text, документы, follow-up answer, business question.
- Стандартизируем metadata: source, timestamp, language, confidence hints.

## Технические шаги
1. Реализовать intake-слой c режимами `analyze` и `dialogue`.
2. Добавить парсер секций, списков, таблиц, числовых блоков.
3. Сформировать `source_manifest.json` с provenance.
4. Сохранять parse-результат в `parsed/<doc_id>.json` и `parsed/<doc_id>.md`.

## Нефункциональные требования
- Устойчивость к частично поврежденным документам.
- Локальные зависимости, без внешних API хранения.
- Логи ошибок читаемы для разработчика в IDE.

## Acceptance
- На тестовом наборе документов уровень parse-failure <10%.
- Для каждого входа создан traceable manifest.
