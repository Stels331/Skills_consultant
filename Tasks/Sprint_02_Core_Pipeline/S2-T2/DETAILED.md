# S2-T2 — Extraction + Typization + Glossary binding (глубокое описание)

## Мета
- Сформульована у задачі та узгоджена з цілями спринта.

## Що робимо
- Виконуємо обсяг робіт, описаний у цій задачі (див. робочі деталі нижче).

## Як виконання
- Реалізація по кроках із фіксацією артефактів у файловій структурі проекту.

## Очікуваний результат
- Отримано завершений артефакт задачі, готовий для інтеграції в потік спринта.

## Як тестувати виконання
- Запустити перевірки/скрипти модуля задачі локально в IDE.
- Перевірити, що всі вихідні файли задачі створені/оновлені коректно.
- Перевірити, що зміни не ламають попередній workflow спринта.

## Контрольні точки поза тестуванням
- Актуальність документації та читабельність артефактів для handoff.
- Відповідність архітектурним правилам (problem-first, evidence-first, no-DB v1).
- Відмітка прогресу задачі та синхронізація зі статусом спринта.

## Робочі деталі (початковий опис задачі)

**Приоритет:** Must


## Scope
- Поддержать AssertionKind: FACT / INTERPRETATION.
- Реализовать aliases, single-type assignment, универсальные relation-ы.
- Генерировать uncertainty-коды при нарушениях правил типизации.

## Технические шаги
1. Сделать extraction pipeline по атомарным утверждениям.
2. Реализовать типизатор с правилами U00..U26 и domain A1..A11 (по приоритету).
3. Добавить glossary resolver и конфликт-детектор контекстов.
4. Сохранять `typed_entities.json` и `type_proposals.json`.

## Нефункциональные требования
- Репродуцируемость extraction на одинаковом входе.
- Минимизация дубликатов сущностей.
- Все результаты хранятся локально в файлах кейса.

## Acceptance
- Не менее 80% ключевых сущностей корректно типизируются на контрольном наборе.
- Все критичные термины имеют привязку к глоссарию.
