## 1. Отобранные решения (Selected)

   * **Обоснование:** Решение прошло Hard Constraints по времени реакции на инцидент. Немедленная остановка ресурсоемких задач (глубокой обработки) критически необходима для снятия нагрузки с системы и предотвращения каскадного отказа.
2. **`sol_02_asynchronous_cyclogram_and_barter`** (Асинхронная циклограмма и бартерный обмен ресурсами)
   * **Обоснование:** Удовлетворяет требованиям к долгосрочной стабильности. Переход от жесткого синхронного планирования к асинхронной циклограмме с механизмом "бартера" (динамического перераспределения квот между потребителями) позволяет утилизировать ресурсы без блокировок.

Решение доминируется предложенными альтернативами по всем ключевым метрикам производительности.

# ADR-001: Переход на асинхронную циклограмму и отключение глубокой обработки

**Статус:** Принято (Accepted)
**Дата:** 2026-03-18
**Workspace ID:** case_20260318_003

## Контекст
Система столкнулась с критическим исчерпанием ресурсов и образованием "бутылочного горлышка" из-за синхронного выполнения задач глубокой обработки. Текущая архитектура (`sol_00_status_quo`) не справляется с нагрузкой, что нарушает SLA и Hard Constraints, описанные в спецификации приемки. Возник конфликт между необходимостью выполнять сложные вычисления и требованием обеспечивать базовую доступность системы в реальном времени.

## Рассмотренные варианты
1. `sol_00_status_quo` — Оставить все как есть.
2. `sol_01_halt_deep_processing` — Полностью остановить модули глубокой обработки.
3. `sol_02_asynchronous_cyclogram_and_barter` — Внедрить асинхронное выполнение задач с динамическим обменом ресурсами (бартером).

## Решение
Принято решение использовать гибридный подход:
1. **Экстренная мера:** Немедленно применить `sol_01_halt_deep_processing` для сброса нагрузки (Load Shedding) и стабилизации ядра системы.
2. **Архитектурная трансформация:** Внедрить `sol_02_asynchronous_cyclogram_and_barter`. Задачи будут выполняться по асинхронной циклограмме, а компоненты системы получат возможность "бартера" — обмена выделенными квотами на CPU/Memory в зависимости от текущего приоритета транзакции.

## Последствия
**Положительные:**
* Мгновенное устранение "бутылочного горлышка" и восстановление доступности системы.
* Повышение отказоустойчивости за счет асинхронной природы новой циклограммы.
* Эффективная утилизация ресурсов благодаря механизму бартера.

**Отрицательные (Компромиссы):**
* Временная (или постоянная) потеря бизнес-ценности от результатов "глубокой обработки".
* Возрастает сложность отладки и мониторинга из-за перехода к асинхронной модели и плавающим квотам (бартеру).

# Runbook: Внедрение и эксплуатация новой архитектуры

## Цель
Обеспечить безопасный перевод системы на асинхронную циклограмму с предварительным отключением тяжелых фоновых процессов.

## Фаза 1: Экстренный сброс нагрузки (sol_01)
1. **Изоляция:** Перевести балансировщик нагрузки в режим `degraded_mode`.
2. **Остановка:** Выполнить команду остановки воркеров глубокой обработки:
   ```bash
   systemctl stop deep-processing-workers
   ```
3. **Очистка очередей:** Сбросить или перенести в Dead Letter Queue (DLQ) текущие задачи глубокой обработки.
4. **Валидация:** Убедиться, что утилизация CPU упала ниже 60%, а время отклика (Latency) вернулось в пределы SLA.

## Фаза 2: Развертывание асинхронной циклограммы (sol_02)
1. **Деплой:** Развернуть новый модуль планировщика (`async-cyclogram-scheduler`).
2. **Конфигурация бартера:** Включить механизм обмена квотами в конфигурации:
   ```yaml
   resource_management:
     mode: "barter"
     max_trade_latency_ms: 50
   ```
3. **Маршрутизация:** Переключить поток задач на новый асинхронный планировщик.
4. **Прогрев:** Плавно увеличивать входящий трафик (по 10% каждые 5 минут).

## Мониторинг (Ключевые метрики)
* `cyclogram.async.queue_depth` — глубина асинхронной очереди (не должна расти экспоненциально).
* `barter.transaction.success_rate` — процент успешных обменов ресурсами (должен быть > 95%).
* `system.bottleneck.io_wait` — ожидание ввода-вывода (должно быть < 5%).

# План отката (Rollback Plan & Blast Radius Mitigation)

## Контекст
Если гипотеза с асинхронной циклограммой и бартером провалится в пилотном цикле (например, возникнет Deadlock при обмене ресурсами или деградация консистентности данных), необходимо безопасно вернуть систему к синхронному состоянию, минимизировав радиус поражения (Blast Radius).

## Условия для инициации отката (Trigger Conditions)
1. Метрика `barter.transaction.success_rate` падает ниже 80% в течение 3 минут.
2. Обнаружен Deadlock в асинхронном планировщике (зависание задач более чем на 30 секунд).
3. Критическая потеря данных из-за отключения глубокой обработки, требующая немедленного возобновления старого процесса.

## Процедура отката (Step-by-Step)
1. **Блокировка новых задач:**
   * Перевести API Gateway в режим `Read-Only` или `Maintenance`.
2. **Отключение бартера и асинхронной циклограммы:**
   * Отправить сигнал завершения (Graceful Shutdown) модулю `async-cyclogram-scheduler`.
   * Принудительно завершить зависшие бартерные транзакции (Kill Switch).
3. **Возврат к Status Quo (sol_00):**
   * Переключить маршрутизацию обратно на старый синхронный планировщик.
   * Запустить воркеры глубокой обработки:
     ```bash
     systemctl start deep-processing-workers
     ```
4. **Рекавери данных:**
   * Повторно отправить задачи из DLQ (Dead Letter Queue), которые были сброшены во время Фазы 1 Runbook'а.
5. **Снятие блокировки:**
   * Вернуть API Gateway в штатный режим работы.

## Митигация радиуса поражения (Blast Radius)
* Использовать Canary-релиз для Фазы 2 (направлять только 5% трафика на асинхронную циклограмму в первые 24 часа).
* Изолировать базу данных бартерных транзакций от основной БД бизнес-логики, чтобы сбой бартера не повредил пользовательские данные.
