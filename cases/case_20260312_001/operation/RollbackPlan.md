# ADR-001: Сохранение текущей архитектуры (Status Quo)

## Контекст
В рамках инициативы по модернизации системы (`case_20260312_001`) были рассмотрены варианты полного переписывания на cloud-native стек (`sol_01`) и закупки готового решения (`sol_02`). Текущая система (`sol_00_status_quo`) обладает накопленным техническим долгом, однако стабильно выполняет бизнес-функции. Нам необходимо было выбрать путь развития, учитывая жесткие ограничения по бюджету, срокам и безопасности.

## Рассмотренные варианты
1. `sol_00_status_quo` (Текущая монолитная/legacy архитектура)
2. `sol_01_cloud_native_rewrite` (Микросервисы)
3. `sol_02_vendor_cots` (Вендорское решение)

## Решение
**Принято решение сохранить `sol_00_status_quo`.** 
Мы отказываемся от масштабной модернизации на данном этапе. Все ресурсы будут перенаправлены на точечный рефакторинг и улучшение observability текущей системы.

## Последствия
### Положительные:
- **Нулевые капитальные затраты (CAPEX):** Бюджет сохранен.
- **Отсутствие миграционных рисков:** Риск простоя из-за неудачной миграции исключен.
- **Предсказуемость:** Команда эксплуатации досконально знает текущую систему.

### Отрицательные:
- **Технический долг:** Сохраняется проблема медленного внедрения новых фич (Time-to-Market для нового функционала остается высоким).
- **Масштабирование:** Требует ручного вмешательства при пиковых нагрузках.

# Эксплуатационный регламент (Runbook) - Status Quo

Так как принято решение остаться на `sol_00_status_quo`, данный Runbook фиксирует правила поддержания текущей системы в условиях отказа от модернизации.

## 1. Мониторинг и Алертинг
- **CPU/RAM Utilization:** Критический порог установлен на 85%. При достижении срабатывает алерт в канал `#ops-legacy-alerts`.
- **Database Connections:** Максимальный пул — 500 соединений. Мониторинг через Grafana (Dashboard ID: `legacy-db-01`).

## 2. Регламент ручного масштабирования (Scale-Out)
Поскольку автоматическое масштабирование ограничено архитектурой:
1. При получении алерта о нагрузке >85% в течение 15 минут, дежурный инженер (L2) запускает скрипт `provision_legacy_node.sh`.
2. Новая нода добавляется в балансировщик (HAProxy) вручную через конфиг `/etc/haproxy/haproxy.cfg`.
3. Выполняется `systemctl reload haproxy`.

## 3. Управление техническим долгом
- Выделение 20% времени спринта (Capacity) команд разработки на точечный рефакторинг самых нагруженных модулей.
- Ежемесячный аудит медленных SQL-запросов (Slow Query Log).

# План отката (Rollback Plan) / Blast Radius Mitigation

*Примечание: Поскольку выбрано решение `sol_00_status_quo`, данный план отката описывает процедуру экстренного сворачивания любых начатых пилотных проектов (Shadow IT, тестовые ветки миграции) и гарантированного возврата всего трафика на стабильную текущую систему.*

## 1. Условия активации плана отката
- Деградация производительности текущей системы из-за влияния пилотных интеграций (Latency > 500ms).
- Ошибки консистентности данных, вызванные тестовой репликацией в новые хранилища.
- Решение стейкхолдеров о полном закрытии экспериментальных веток.

## 2. Оценка радиуса поражения (Blast Radius)
- **Затронутые компоненты:** Балансировщик трафика, брокер сообщений (Kafka/RabbitMQ), реплики БД.
- **Риск потери данных:** Низкий, так как пилотные системы работали в режиме Read-Only или Shadow Traffic.

## 3. Пошаговый план отката (Revert to Status Quo)

### Шаг 1: Остановка теневого трафика (Shadow Traffic)
- Отключить зеркалирование трафика на уровне API Gateway / Nginx:
  ```bash
  # Закомментировать директиву mirror в nginx.conf
  sed -i 's/^\s*mirror /#mirror /g' /etc/nginx/nginx.conf
  nginx -s reload
  ```

### Шаг 2: Изоляция экспериментальных консьюмеров
- Отключить консьюмеры пилотных систем от production-топиков брокера сообщений, чтобы снизить нагрузку на сеть и диск.

### Шаг 3: Очистка и блокировка (Sanitization)
- Удалить экспериментальные индексы и триггеры из production БД, которые были созданы для поддержки пилотов.
- Заблокировать сетевые доступы (Firewall/Security Groups) от пилотных кластеров к production БД.

### Шаг 4: Верификация стабильности (Status Quo Verified)
- Проверить метрики в Grafana: Latency должно вернуться к норме (< 200ms), Error Rate < 0.1%.
- Провести прогон E2E smoke-тестов для подтверждения полной работоспособности `sol_00_status_quo`.
